# Firmware Architecture Patterns: Structuring Embedded Software for Scale

You have just been handed a codebase for a motor drive controller. The original developer left the company. There are no comments, no header files with meaningful declarations, and a single 3,000-line main.c that reads SPI directly in the middle of the control loop, writes to GPIO registers inside the UART receive interrupt, and uses thirty-seven global variables with names like flag2 and temp_val. Adding a new feature takes two weeks because every change breaks something unrelated. Porting to the new MCU variant the hardware team just selected will require touching every file. Tests are nonexistent because nothing can be tested without the physical board powered up. This codebase is not an extreme example -- it is what naturally happens when firmware grows without a deliberate architectural plan.

This article is about how to prevent that outcome by choosing and applying a deliberate firmware architecture from the start. It covers the three-layer model that separates hardware from application logic, the hardware abstraction layer (HAL) and what makes a good one, device drivers and how they differ from the HAL, the board support package (BSP) and its role as the hardware-specific glue, component-based partitioning of application logic, the dependency rules that keep layers decoupled, interface design principles that enable portability and testability, and the anti-patterns that appear most often in production firmware that has grown beyond its original design. The target platform throughout is ARM Cortex-M, but the principles apply equally to any microcontroller family.

## The Core Problem

Ad-hoc firmware works fine when a project has one developer, one MCU variant, one set of features, and a short lifetime. As soon as any of those conditions changes, the lack of architecture begins to impose a compounding tax. Tight coupling is the root cause. When application logic calls peripheral registers directly, a change to the hardware requires changes throughout the application. When drivers assume a specific MCU, porting to a new silicon vendor means rewriting logic that has nothing to do with hardware. When modules share state through global variables, a bug in one module corrupts data owned by another, and the failure appears far from the root cause. When initialization order is implicit rather than explicit, changing the order of two function calls produces a crash that takes days to diagnose. Each of these problems is a form of coupling: two pieces of code that should be independent are instead entangled, so a change to one forces a change to the other.

The cost of tight coupling compounds over time because it makes every future change more expensive. A firmware codebase that starts with no architecture does not stay small. Features accumulate, hardware revisions arrive, new engineers join and write in whatever style the existing code implies. Without architectural boundaries, the complexity of the system grows as the square of the number of components rather than linearly. Engineers in this situation spend more time understanding the existing code than writing new features. Bug fix rates slow as the blast radius of any change grows. At some point the codebase becomes the dominant obstacle to shipping the product. A deliberate firmware architecture is the investment that prevents this trajectory.

## Architectural Layers

The three-layer model is the most widely applied architecture in production embedded firmware. It divides the codebase into an application layer at the top, a middleware layer in the middle, and a hardware layer at the bottom. Each layer is allowed to call downward into the layer below it. No layer is permitted to call upward. This single rule -- upper layers depend on lower layers, never the reverse -- is the load-bearing constraint of the entire architecture. It is the rule that makes the system portable, testable, and maintainable.

The hardware layer consists of the hardware abstraction layer (HAL) and the board support package (BSP). Its responsibility is to present the rest of the firmware with a stable, MCU-independent interface to the hardware. The application layer has no knowledge of whether the SPI peripheral is driven by a Cortex-M4 DSPI module or an STM32 SPI peripheral; it only knows that it can call spi_transfer() and receive data. The middleware layer sits between application and hardware and contains reusable components that are hardware-independent but not application-specific: protocol stacks, data encoders, ring buffers, filter algorithms, and similar building blocks. The application layer contains the product-specific logic that determines what the firmware actually does. Interface contracts between layers -- the function signatures and behavioral guarantees exposed by each layer to the one above -- are the joints that hold the system together. Defining these contracts explicitly, before writing implementations, is the primary design activity in firmware architecture.

## The Hardware Abstraction Layer (hal)

The hardware abstraction layer is a collection of API functions that expose peripheral capabilities -- serial buses, GPIO, timers, ADC, DMA -- without exposing any hardware-specific register details. Its purpose is to place a stable boundary between software that needs a capability and hardware that provides it. Above the HAL, code is written in terms of capabilities: "transfer eight bytes over SPI." Below the HAL, code is written in terms of silicon: "write 0x07 to SPI_CR1, poll until TXE is set, write the byte to SPI_DR." When the hardware changes, only the below-the-HAL code needs to change. Everything above the HAL is unaffected. This is the MCU-independence principle: the application layer and middleware layer are not written for a specific MCU; they are written for the HAL interface, which happens to be implemented on top of a specific MCU.

A well-designed HAL API has a consistent structure across all peripherals. An init() function accepts a configuration structure and prepares the peripheral for use; it returns a status code, never void. A read() or receive() function retrieves data from the peripheral into a caller-supplied buffer. A write() or transmit() function sends data to the peripheral from a caller-supplied buffer. A status() function returns current peripheral state without performing any operation. A deinit() or close() function releases the peripheral and returns it to a known idle state. All functions return an explicit status type -- an enum or integer -- that the caller is expected to check. Functions that can fail must never return void. The HAL does not own data buffers; callers provide them. The HAL does not perform application logic; it transfers data between buffers and hardware. These constraints keep the HAL thin, predictable, and easy to port.

## Device Drivers

A device driver is software that manages a specific external component -- a sensor, actuator, memory chip, or display -- by issuing a sequence of HAL calls. The HAL abstracts a peripheral bus; the driver abstracts a component. An I2C temperature sensor driver knows the device's register map, its initialization sequence, its conversion timing, and its error conditions. The application layer does not know any of these details; it only calls temperature_read(&result). The driver translates that call into the sequence of I2C HAL calls required to retrieve a raw measurement and convert it to engineering units. This separation means the driver can be reused on any MCU that provides the same I2C HAL interface, and the application can swap one temperature sensor for a different one by replacing only the driver.

The canonical example of driver-over-HAL architecture is a SPI NOR flash driver sitting on top of a SPI HAL. The flash device has a specific command set: 0x03 for Read, 0x02 for Page Program, 0x20 for Sector Erase, 0x05 for Read Status Register. The driver sequences these commands: to read a page it asserts chip select via the GPIO HAL, transmits the Read command and 24-bit address via the SPI HAL, clocks in the data, then deasserts chip select. To write, it must first issue a Write Enable command, wait for the previous operation to complete by polling the status register, then issue the Page Program command. The driver owns this protocol knowledge entirely. The SPI HAL knows nothing about flash command sets. The application knows nothing about SPI transfers. Each layer does one thing and exposes a clean interface to the layer above.

## The Board Support Package (bsp)

The board support package is the bridge between a specific printed circuit board and the generic HAL interface. If the HAL defines what a GPIO or SPI peripheral looks like in the abstract, the BSP defines where each GPIO pin is connected on this specific board, which SPI peripheral serves which device, and how the clocks are configured on this specific MCU at this specific frequency. The BSP is the only layer that is allowed to contain board-specific constants: pin numbers, alternate function selections, clock divider values, interrupt routing assignments. When the hardware team spins a new board revision that moves a sensor from SPI1 to SPI2 or changes a GPIO pin from PA3 to PB7, only the BSP changes. The HAL implementation and the drivers above it are untouched.

Typical contents of a BSP include the system clock configuration function that sets PLL multipliers, AHB and APB prescalers, and flash wait states for the target operating frequency; GPIO initialization that sets direction, pull configuration, alternate function, and output speed for every pin on the board; interrupt routing that assigns peripheral interrupt lines to NVIC channels and sets priority levels; a board identifier constant that higher-level code can query if it needs to branch on hardware variant; and a board_init() function that calls all of the above in the correct order. The BSP is compiled once per board variant. Switching between board variants is a matter of selecting which BSP source files are included in the build, not modifying driver or application code.

## Component-Based Architecture

As firmware grows beyond a few thousand lines, a flat three-layer model with large monolithic source files in each layer becomes unmanageable. Component-based architecture extends the layered model by further partitioning each layer into independent components, each of which owns a single concern, exposes an explicit interface, and has no direct dependencies on the internal state of other components at the same layer. A component is typically a .c/.h pair: the header declares the public API, and the source file contains all private state as static variables and all implementation detail as static functions. Other components at the same layer can only interact with it through the public API. This discipline is the C-language equivalent of encapsulation in object-oriented design, and it provides the same benefits: isolation, testability, and replaceability.

Initialization ordering is one of the practical challenges in component-based firmware. Components have dependencies -- a UART driver must be initialized before a console logging component that uses it -- but if these dependencies are encoded implicitly through the order of calls in main(), they become fragile. Any future reordering breaks the firmware silently. The explicit approach is to give each component an init() function that asserts or checks that its dependencies are initialized before proceeding, and to document the dependency chain in the component's header. An alternative is dependency injection: rather than having a component reach out and call its dependency's HAL directly, the caller passes a pointer to the HAL interface at initialization time. This makes the dependency visible in the API and allows the same component to be initialized with a mock interface during unit testing, eliminating the need for the physical hardware to be present.

## Layered Dependency Rules

The rule that upper layers call lower layers and lower layers never call upper layers is not just a stylistic preference; it is a structural property that determines whether the firmware is testable and portable. If a driver calls a function in the application layer, the driver can no longer be tested in isolation -- testing the driver requires instantiating the application. If the HAL calls a driver, the HAL becomes coupled to specific devices, and porting it means rewriting device-specific logic. Violations of the dependency rule always manifest as maintenance problems: the codebase becomes a graph of circular dependencies, nothing can be compiled without compiling everything, and nothing can be tested without testing everything.

Callbacks are the mechanism by which lower layers legitimately signal events upward without creating an upward dependency. A driver registers a callback pointer at initialization time -- the application passes a function pointer to the driver's init() function, and the driver calls it when an event occurs. The driver depends only on the function signature, not on the specific function or the module it belongs to. This pattern is used universally in embedded firmware for DMA completion callbacks, UART receive callbacks, GPIO edge callbacks, and timer expiry callbacks.

## Interface Design Principles

An opaque handle is the C pattern for encapsulating component state behind an interface. Instead of exposing a struct with named fields that callers can access directly, the component declares a handle as a pointer to a forward-declared (incomplete) struct type. The struct definition lives only in the .c file. Callers hold the handle and pass it to API functions, but they cannot read or write the struct's fields directly. This prevents callers from depending on the internal layout of the component, which means the internals can change without breaking the callers. STM32 HAL's HAL_SPI_HandleTypeDef is a well-known counter-example of this principle: exposing all internal fields forces application code to initialize them directly, creating coupling between the application and the HAL's internal structure.

Function pointer tables implement polymorphism in C without virtual dispatch overhead. A driver interface is defined as a struct containing function pointers: one for init, one for read, one for write, one for status, one for deinit. Each concrete driver (say, two different SPI flash devices from different vendors) populates a static const instance of this struct with its own function implementations. The application layer holds a pointer to the interface struct and calls through the function pointers without knowing which concrete driver is behind them. This pattern is how the HAL achieves MCU independence: there is one HAL interface struct per peripheral type, and each MCU port fills it in with the correct register-level implementations. The callback registration pattern is similar: a component exposes a register_callback() function that stores a function pointer and an optional context pointer; when the event occurs, the component calls the stored function pointer with the context pointer as an argument, allowing the caller to recover its own state without the component knowing anything about it.

## Porting and Testability

The practical payoff of a clean layered architecture appears most clearly in two scenarios: porting to a new MCU and unit testing without hardware. Porting is reduced to swapping the BSP and the HAL implementations. Everything above the HAL -- drivers, middleware, application -- is recompiled without modification against the new HAL implementation. In practice, some drivers will need minor adjustments if they relied on timing assumptions specific to the original MCU, but the scope of change is bounded and auditable. Compare this to porting a codebase with register-level code scattered through the application layer: every file must be audited for hardware-specific assumptions, and there is no systematic way to know when the port is complete.

Unit testing without hardware requires that the code under test has no direct dependencies on hardware registers. In a well-layered architecture, the driver layer and above have no such dependencies -- they call only HAL functions. For testing, the HAL functions are replaced with mock implementations generated by a tool like CMock in the Ceedling framework. The mock records which HAL functions were called, with what arguments, and in what order; it also allows the test to inject return values that simulate specific hardware conditions. A SPI flash driver can be tested by mocking the SPI HAL and the GPIO HAL, injecting a sequence of simulated read-back bytes, and asserting that the driver correctly interprets a busy status, a ready status, and a data payload without a physical flash device or MCU in the loop. This test suite runs on the development workstation in milliseconds and catches protocol logic bugs before the firmware is ever loaded onto hardware.

## Common Anti-Patterns

The god module is a source file that accumulates responsibility for everything: it initializes hardware, runs the main control algorithm, handles communication, manages error recovery, and maintains global state for every other module. God modules form because they are the path of least resistance when adding a feature: find the one place where everything is accessible and add the new code there. The cost is that god modules cannot be tested in isolation, cannot be reused, and cannot be maintained by anyone who was not there when they were written. The remedy is continuous decomposition: any time a source file exceeds a few hundred lines, ask what single concern it should own and extract everything else into a new component.

Register-banging scattered through the application layer is the failure mode where hardware register writes appear in files that are supposed to contain application logic. A product manager reads a spec change that requires a different UART baud rate; the engineer searching for the UART configuration finds it split across three application files because the original developer wrote USART2->BRR = 0x1A1 wherever they needed the UART to work. Fixing it requires understanding all three call sites, their interactions, and their startup ordering. The remedy is to enforce that no file above the HAL layer contains any register access, peripheral address, or vendor-specific peripheral type. Code review is the enforcement mechanism; static analysis rules (such as a MISRA C check banning direct peripheral struct dereferences outside designated BSP files) can automate it.

Global state spaghetti occurs when components communicate by reading and writing shared global variables rather than through function interfaces. The symptom is a bug that only appears when two modules execute in a particular interleaved order, because one module reads a global that the other module wrote in a partially updated state. The remedy is to make all inter-component communication explicit: pass data as function arguments, return it as function return values, or use a dedicated event queue. Static variables at file scope that are private to a single component are acceptable; global variables accessible by multiple components are a design flaw.

ifdef hell is the anti-pattern where hardware-specific and product-variant-specific code is conditionally compiled throughout the codebase using preprocessor conditionals. A file that contains twenty nested ifdefs for different board variants, different MCU families, and different feature configurations cannot be read linearly; understanding what code is active requires mentally simulating the preprocessor with a specific set of defines. The remedy is to isolate variant-specific code in separate source files selected at build time, rather than inline conditionals. Each board variant has its own BSP source file; each MCU port has its own HAL implementation file; the build system selects the correct files for the target. Source files contain linear, unconditional code. Ifdefs are reserved for header guards and for toggling optional features that are truly orthogonal.

## Real-World Example

A concrete project structure illustrates how these principles map to a real directory layout. Suppose the project is a battery management system (BMS) firmware for an STM32G4 MCU. The top-level directories are: src/app/ for application logic, src/middleware/ for reusable non-hardware components, src/drivers/ for device-specific drivers, src/hal/ for the HAL interface headers and the STM32G4 HAL implementation, and src/bsp/ for the board-specific pin assignments and clock configuration for this specific BMS board revision. The hal/ directory contains one subdirectory per peripheral type: hal/spi/, hal/i2c/, hal/uart/, hal/gpio/, hal/timer/. Each subdirectory contains an interface header (spi.h defining the SPI HAL API) and an implementation source (spi_stm32g4.c containing the register-level code). A port to a different MCU adds spi_nxps32k.c alongside the existing file; the build system selects the correct one.

The drivers/ directory contains one subdirectory per external device: drivers/bq76952/ for the Texas Instruments BQ76952 battery monitor IC, drivers/m25p80/ for a SPI NOR flash. Each driver directory contains a header exposing the driver's public API (bq76952.h declaring bq76952_init(), bq76952_read_voltage(), bq76952_read_current(), bq76952_set_alert_threshold()) and an implementation source. The driver implementation includes only the HAL headers it needs; it never includes a BSP header or an application header. The app/ directory contains modules named by concern: app/battery_monitor.c, app/charge_controller.c, app/fault_manager.c, app/comm_handler.c. Each module's header exposes an init() and a run() function and any data structures the calling code needs to interact with it. Header discipline is enforced: every header includes only the headers it directly needs, uses include guards, and declares no global variables -- all state is encapsulated in static variables in the .c file.

## Choosing Your Architecture

The three-layer model with component-based partitioning is not the correct choice for every project. The cost of the architecture is upfront: defining interfaces before writing implementations requires more thought at the start, and the additional file structure adds build system complexity. For a one-engineer project on a single MCU with a short lifetime and no reuse requirements, a simpler flat structure may be appropriate. The decision criteria that justify the full layered architecture are: team size greater than one (because multiple engineers need defined boundaries to work in parallel without stepping on each other); reuse goals (if any driver, middleware component, or application module is planned for use in more than one product); target complexity (if the firmware will exceed roughly 10,000 lines of code, the maintenance cost of a flat structure begins to outweigh the upfront cost of defining interfaces); and compliance requirements (safety standards such as IEC 61508 or ISO 26262 require traceability from requirements to code to tests, which is far easier when the code is partitioned into components with explicit interfaces).

An RTOS changes the architecture in specific ways but does not change the layering principle. With an RTOS, the application layer is partitioned into tasks rather than cooperative module run() functions, and inter-component communication uses RTOS queues and semaphores rather than direct function calls and flags. The HAL, BSP, and driver layers remain structurally identical. The key architectural decision with an RTOS is which layer owns the RTOS abstractions: the answer is that RTOS API calls belong in the application layer and in middleware components that are explicitly RTOS-aware. Drivers and the HAL remain RTOS-agnostic; they should be callable from both task context and ISR context, with the distinction managed by the driver's internal locking strategy. A driver that takes a mutex internally is not portable to a bare-metal system; a driver that accepts a lock/unlock function pointer pair at initialization time is portable to both.

## Key Takeaways

The three-layer model -- application, middleware, hardware -- is the foundational structure that separates what the firmware does from how it does it on a specific piece of silicon, and this separation is what makes firmware portable and testable. The HAL defines a stable interface between capability-level code and register-level code; its anatomy follows a consistent init/read/write/status/deinit pattern with explicit error returns on every function. The BSP is the only legitimate location for board-specific constants and pin assignments; isolating it here means a board revision or MCU swap touches one directory, not every source file. Component-based partitioning within each layer enforces single-concern modules, encapsulated state, and explicit interfaces, which together make it possible to unit test firmware logic on a workstation without hardware. The common anti-patterns -- god modules, register-banging in application code, global state spaghetti, ifdef hell -- are all forms of coupling that violate layer boundaries; recognizing them early and enforcing boundaries through code review and static analysis is the maintenance discipline that keeps firmware manageable as it grows.

## Further Reading

Jacob Beningo's "Reusable Firmware Development" (Apress, 2017) is the primary reference for the HAL and component-based patterns described in this article; it provides C code examples for Cortex-M that directly illustrate the init/read/write/status/deinit API structure and the opaque handle pattern. Beningo's companion volume "Reusable Device Drivers" goes deeper on the driver layer, covering protocol sequencing, error handling, and the function-pointer table pattern for polymorphic drivers in C. The Embedded Artistry blog (embeddedartistry.com), maintained by Phillip Johnston, contains an extensive series of articles on HAL design, abstraction patterns, and firmware architecture decisions, with particular depth on interface design and testability. The Barr Group Embedded C Coding Standard (available at barrgroup.com) provides the naming conventions, file structure rules, and module organization guidelines that, applied consistently across a team, produce the uniform house style that makes a codebase read as if written by one engineer. For engineers working on safety-critical products, the MISRA C:2025 guidelines (Motor Industry Software Reliability Association) provide the rule set for C language usage that, when combined with the architectural patterns in this article, produces firmware that satisfies the code quality requirements of IEC 61508 and ISO 26262 functional safety standards.
