# CPU Execution Model: How a Cortex-M Processor Executes Instructions

A firmware engineer is chasing a hard fault that appears only at 168 MHz and vanishes when the debugger is attached. The stack frame looks corrupted, the LR value points nowhere meaningful, and the fault address is inside what should be a safe read. Without a solid mental model of how the Cortex-M processor fetches, decodes, and executes instructions, how it manages its register bank, how exception entry constructs a stack frame, and how the pipeline interacts with memory latency, debugging this class of problem becomes guesswork. This article builds that mental model from the ground up, starting with the three-stage pipeline and ending with the memory barriers that separate correct firmware from firmware that works by accident on a particular silicon revision.

The ARM Cortex-M family (M0, M0+, M3, M4, M23, M33, M55) all share the same fundamental programmer model even though they differ in pipeline depth, DSP capability, and security extensions. Everything in this article applies to the M3 and M4 unless a specific difference is called out. STM32F4, STM32F7, STM32H7, STM32G4, and STM32L5 all use cores in this family, and the principles transfer directly to nRF52, SAM, and i.MX RT microcontrollers as well.

## The Fetch-Decode-Execute Pipeline

The Cortex-M3 and M4 implement a three-stage in-order pipeline: fetch, decode, and execute. In the fetch stage the processor reads one or more 16-bit halfwords from instruction memory at the address held in the Program Counter (PC, also called r15). In the decode stage the processor determines the instruction type, the source and destination registers, and any immediate values encoded in the binary. In the execute stage the Arithmetic Logic Unit (ALU) performs the operation, the memory interface issues any load or store, and the result is written back to the destination register or memory location. At any given moment all three stages are active simultaneously on three consecutive instructions, so in the steady state (no branches, no stalls) one instruction completes every clock cycle on a zero-wait-state memory system.

A branch instruction breaks pipeline continuity because the processor has already fetched and begun decoding the two instructions after the branch before it knows whether the branch is taken. On the Cortex-M3 and M4 a taken branch incurs a three-cycle penalty while the pipeline refills from the branch target address. The processor does not implement dynamic branch prediction in the way a desktop CPU does; it has a simple static scheme where a backward branch (target address lower than branch instruction address) is predicted taken and a forward branch is predicted not-taken. This is why tight inner loops that branch backward are efficient, and why placing frequently executed code in a linear sequence without forward branches gives the best throughput. If you are working on an STM32H7 or another part with an M7 core (which is not a Cortex-M core but often grouped with the family), the M7 has a six-stage dual-issue superscalar pipeline and a more sophisticated branch predictor, making these tradeoffs more complex.

The pipeline also interacts with the bus interface. On a Cortex-M the instruction fetch and data access share the same AHB (Advanced High-performance Bus) matrix but use separate bus masters (I-Code and D-Code for internal flash, and S-Bus for SRAM). A data-intensive instruction that accesses memory in the same cycle as the next instruction fetch must arbitrate for bus bandwidth. Understanding this explains why alternating load/store sequences with ALU-only instructions can hide memory latency and improve throughput on memory-constrained designs.

## Registers and the Programmer Model

The Cortex-M programmer model exposes 16 registers in the integer register bank, all 32 bits wide. Registers r0 through r12 are general-purpose registers available for integer arithmetic, addresses, and temporary storage. Of these, r0 through r3 are designated as argument and scratch registers by the ARM Procedure Call Standard (AAPCS): the first four integer or pointer arguments to a function are passed in r0-r3, and return values come back in r0 (or r0:r1 for 64-bit values). The compiler is free to use r0-r3 for any intermediate computation within a function without saving and restoring them, because the calling convention does not require their preservation across a call. Registers r4 through r11 are callee-saved: if a function uses them it must push them onto the stack at entry and pop them at exit, guaranteeing that the caller sees their original values after the call returns.

Register r12 (also named IP, the intra-procedure-call scratch register) is technically caller-saved and is sometimes used by linker veneers for long-range branches; firmware code can use it freely as a scratch register with the same caveats as r0-r3. Register r13 is the Stack Pointer (SP). The Cortex-M implements two stack pointers, the Main Stack Pointer (MSP) and the Process Stack Pointer (PSP), but only one is active at any time; in bare-metal firmware running entirely in privileged Thread mode you will almost always use only the MSP. Register r14 is the Link Register (LR). When a BL (Branch with Link) instruction executes, the processor writes the return address (the address of the instruction after BL) into LR before jumping to the target. A leaf function -- one that calls no other functions -- can return simply by branching to LR without touching the stack. Register r15 is the Program Counter (PC). Reading r15 in user code returns the address of the current instruction plus 4 (a legacy of the original ARM pipeline); writing r15 causes a branch. On Cortex-M, PC bit 0 must always be 1 when written via BX or POP because Thumb mode is mandatory; writing an even address causes a UsageFault.

The Cortex-M4 additionally exposes 32 floating-point registers (s0-s31, also accessible as d0-d15 for double-precision pairs and q0-q7 for 128-bit NEON-style vectors) when the optional FPU is implemented and enabled. The FPU has its own calling convention considerations: s0-s15 are caller-saved and s16-s31 are callee-saved. Lazy FPU stacking on exception entry is an important optimization discussed in the exceptions section.

## The Program Status Register

The xPSR (combined Program Status Register) is a 32-bit register that the processor updates automatically on every instruction. It is not directly addressable as a single register; instead it is composed of three overlapping views: the Application Program Status Register (APSR), the Execution Program Status Register (EPSR), and the Interrupt Program Status Register (IPSR). The MRS and MSR instructions read and write xPSR fields from software.

The APSR holds the condition flags in its upper bits: N (bit 31, set when the result of an arithmetic or logical operation is negative), Z (bit 30, set when the result is zero), C (bit 29, set when an unsigned arithmetic operation produces a carry or a shift operation produces a carry-out), and V (bit 28, set when a signed arithmetic operation overflows). On the Cortex-M4 the APSR also holds Q (bit 27, the saturation flag for DSP-style saturating arithmetic) and the GE bits (bits 19:16, used by parallel add/subtract SIMD instructions). These flags are only updated when the instruction has an "S" suffix (e.g., ADDS, SUBS, MOVS) or when the instruction always updates flags by definition (e.g., CMP, TST). Conditional branch instructions (BEQ, BNE, BCS, etc.) evaluate these flags to decide whether to take the branch.

The EPSR holds the Thumb execution state bit (T bit, bit 24). On Cortex-M this bit must always be 1 because the core has no ARM instruction set mode. It also holds the IT (If-Then) state bits and, on cores with hardware divide or specific instructions, additional state. The IPSR holds the current exception number in bits 8:0. A value of zero means the processor is in Thread mode; a non-zero value identifies which exception handler is currently executing (1 = Reset, 2 = NMI, 3 = HardFault, 4-15 = fixed system exceptions, 16+ = IRQ0 and above). Firmware can read the IPSR to determine whether it is running in an interrupt context without relying on RTOS APIs.

## Instruction Set Overview

Cortex-M processors execute the Thumb-2 instruction set, a superset of the original 16-bit Thumb instruction set introduced with ARM7TDMI. Thumb-2 is a mixed-width encoding: most common operations (arithmetic on low registers r0-r7, short branches, load/store with small offsets) are encoded in 16 bits, while operations requiring full register access, wide immediate values, or complex addressing modes use 32-bit encodings. The processor distinguishes 16-bit from 32-bit instructions by inspecting bits 15:11 of the first halfword; values 11101, 11110, and 11111 indicate a 32-bit instruction follows, and anything else is a complete 16-bit instruction.

The original ARM architecture used a fixed 32-bit instruction encoding with a 4-bit condition code field on every instruction, allowing any instruction to be conditionally executed. Thumb-2 replaces most per-instruction conditioning with the IT (If-Then) block, a 16-bit instruction that conditions the next one to four instructions based on a flag test. The compiler typically generates IT blocks automatically; hand-written assembly should use them correctly to avoid UNPREDICTABLE behavior, which on silicon usually means the IT state machine gets out of sync and the processor executes the wrong instructions. The ARMv8-M architecture (Cortex-M23 and M33) removes the IT block from many contexts in Thumb-2 to simplify security analysis, so firmware written for M33 should be aware that the assembler may reject IT blocks that were accepted on M3/M4 targets.

Instruction density -- the ratio of useful operations to bytes of instruction memory consumed -- is better with Thumb-2 than with fixed-width ARM32 for typical C-compiled code. Studies cited in Yiu's "Definitive Guide to Cortex-M" consistently show a 25-35% code size reduction compared to ARM32 for the same C source. For embedded systems where flash is measured in kilobytes, this matters enormously. The tradeoff is that the variable-width encoding makes static analysis of instruction boundaries harder (you cannot simply walk memory in fixed 4-byte steps), which is relevant for stack unwinding, code coverage tools, and any analysis that walks the instruction stream.

## Memory-Mapped Architecture

The Cortex-M memory map is flat and unified: every peripheral register, every SRAM bank, every flash sector, every debug register, and every core peripheral (NVIC, SysTick, SCB, FPU control) lives at a specific 32-bit address, and the same load/store instructions that read a variable from RAM are used to read a peripheral status register. There is no separate I/O address space, no IN/OUT instructions, and no distinction between "memory" and "device" at the instruction level -- the distinction is made by the Memory Protection Unit (MPU) attribute tables and by the bus fabric routing that connects an address range to a particular slave interface.

The ARMv7-M memory map divides the 4 GB address space into fixed regions with default memory attributes. Addresses 0x00000000-0x1FFFFFFF are Code (typically flash, with instruction and data caching allowed). Addresses 0x20000000-0x3FFFFFFF are SRAM (write-through or write-back cacheable, always executable unless the XN attribute is set). Addresses 0x40000000-0x5FFFFFFF are Peripheral (device memory, non-cacheable, non-executable). Addresses 0xE0000000-0xFFFFFFFF are System (the private peripheral bus, PPB, containing the NVIC at 0xE000E000, SysTick at 0xE000E010, SCB at 0xE000ED00, MPU at 0xE000ED90, and CoreDebug at 0xE000EDF0). These addresses are the same on every Cortex-M part regardless of silicon vendor; the STM32F4 NVIC is at the same address as the nRF52 NVIC, making the core peripheral driver code fully portable.

Access width matters at the hardware level. Most peripheral registers are 32 bits wide and must be accessed as 32-bit words; byte or halfword reads may return only the accessed portion, and byte or halfword writes to a 32-bit peripheral register produce UNPREDICTABLE results (the peripheral may ignore the write, may write only the accessed bytes, or may write garbage to the remaining bytes). The compiler generates the correct access width when you use volatile uint32_t pointers for peripheral registers; using uint8_t pointers to peripheral registers is a common and subtle bug that works on some peripherals by accident but fails on others.

## The Stack and Function Calls

The Cortex-M uses a full descending stack model: the stack grows from high addresses toward low addresses, and the stack pointer always points to the last occupied slot (the "full" convention, as opposed to an empty convention where SP points to the next free slot). A PUSH {r4, lr} instruction decrements SP by 8 (4 bytes per register), then writes r4 to SP+4 and LR to SP, in that order. A POP {r4, pc} reads SP into r4, reads SP+4 into PC, then increments SP by 8. Writing the popped value directly to PC (rather than to LR followed by BX LR) is the standard return mechanism for non-leaf functions.

The function prologue and epilogue generated by a C compiler for an AAPCS-compliant function are predictable: the prologue PUSHes any callee-saved registers the function uses, and the epilogue POPs them back. The frame pointer (r7 on Thumb, r11 on ARM32) is optional; GCC omits it by default for Thumb-2 targets when compiled with optimization, using SP-relative addressing for local variables instead. Omitting the frame pointer is correct and saves a register, but it makes manual stack unwinding harder because the debugger must rely on DWARF unwind information rather than a chain of frame pointers. For production firmware, ensure your linker command line includes -Wl,--eh-frame-hdr or equivalent so that unwind tables are emitted; without them the debugger cannot reconstruct the call chain through functions without a frame pointer.

On exception entry the processor automatically pushes an 8-word exception frame onto the active stack: in order from lower to higher address, xPSR, PC (the instruction address where execution was interrupted), LR (the return address at the point of interruption), r12, r3, r2, r1, r0. The values are written in that order by the hardware, independent of any software action. This automatic stacking means an interrupt service routine (ISR) compiled as a normal C function can use r0-r3 and r12 freely without saving them; the hardware has already saved them on the stack. If the FPU is active (FPCCR.LSPEN is set), the processor also reserves 18 additional words for the floating-point state, but uses lazy stacking to defer the actual write until the ISR first accesses the FPU, avoiding the cost on ISRs that never use floating-point.

## Operating Modes and Privilege

The Cortex-M processor operates in one of two modes and one of two privilege levels at any given time. Thread mode is the normal execution context; code runs in Thread mode when not handling an exception. Handler mode is entered automatically on exception entry; all exception handlers, including all interrupt service routines, run in Handler mode. Handler mode is always privileged. Thread mode may be either privileged or unprivileged, controlled by the nPRIV bit in the CONTROL register.

Privileged code can read and write all processor registers and access the full memory map. Unprivileged code is subject to MPU restrictions and cannot access certain system registers (CONTROL, PRIMASK, BASEPRI, FAULTMASK, MSP, PSP, and the core peripheral registers). A RTOS typically runs application tasks in unprivileged Thread mode and uses the MPU to confine each task to its own stack and heap region; a task that overruns its stack or corrupts a neighbor's memory triggers an MPU fault (MemManage fault) rather than silently corrupting system state. Bare-metal firmware running without an OS almost never leaves privileged Thread mode; the added complexity of privilege transitions is rarely justified unless the firmware is implementing a security boundary.

The CONTROL register also selects which stack pointer is active in Thread mode. When CONTROL.SPSEL = 0, Thread mode uses the Main Stack Pointer (MSP). When CONTROL.SPSEL = 1, Thread mode uses the Process Stack Pointer (PSP). Handler mode always uses the MSP. A typical RTOS implementation initializes each task with its own stack and sets CONTROL.SPSEL = 1 before launching tasks, so that the RTOS kernel and interrupt handlers share the MSP (which is sized for ISR nesting) while each task has its own PSP-based stack. This separation means a task stack overflow corrupts only the PSP region; the kernel stack (MSP) is unaffected, allowing the RTOS to detect and handle the overflow gracefully.

## Processor Exceptions and the Nvic

The Cortex-M exception model unifies interrupts, faults, and system events into a single numbered exception space. Exception numbers 1 through 15 are fixed system exceptions: 1 (Reset), 2 (NMI), 3 (HardFault), 4 (MemManage), 5 (BusFault), 6 (UsageFault), 7-10 (reserved), 11 (SVCall), 12 (DebugMonitor), 13 (reserved), 14 (PendSV), 15 (SysTick). Exception numbers 16 and above map to external interrupts IRQ0, IRQ1, etc., with the exact number available depending on the silicon vendor's NVIC configuration (STM32F4 implements 82 maskable interrupts, so exceptions 16-97 are defined).

The Nested Vectored Interrupt Controller (NVIC) manages priority, pending, and active state for all external interrupts. Priority is configured per-interrupt with 3 to 8 bits of priority field depending on the part (STM32 uses 4 bits, giving 16 software-configurable priority levels). The NVIC implements priority grouping via the AIRCR.PRIGROUP field, which splits the priority bits into preemption priority and sub-priority; two interrupts with the same preemption priority cannot preempt each other (tail-chaining instead), but sub-priority determines which executes first when both are pending simultaneously. The three highest-priority exceptions (Reset, NMI, HardFault) have fixed negative priority values and cannot be masked by software.

Exception entry sequence: when the processor accepts an exception, it simultaneously pushes the 8-word stack frame described earlier and fetches the exception vector from the vector table (base address configured by the VTOR register in the SCB). The vector table entry holds the address of the handler function with bit 0 set to 1 (Thumb mode indicator). After stacking, the processor sets the IPSR to the exception number, switches to Handler mode, and begins executing the handler at the fetched address. Exception return is triggered by loading a special EXC_RETURN value into the PC; the processor recognizes addresses of the form 0xFFFFFFXX as EXC_RETURN magic values rather than real addresses. The EXC_RETURN value encodes which stack was in use before the exception (MSP or PSP) and whether the processor was in Thread or Handler mode, so the return restores the correct state.

## Wait States and Execution Timing

Flash memory on STM32 and similar microcontrollers cannot sustain the read rate demanded by a processor running at maximum clock frequency. On the STM32F4 running at 168 MHz with 3.3 V supply, the ART Accelerator (Adaptive Real-Time cache) is required to deliver zero-wait-state performance from flash; without it the processor would incur 5 wait states per instruction fetch, reducing effective throughput to roughly 168/6 = 28 MIPS. The ART Accelerator is a dedicated 64-bit wide instruction cache (128 bit on F7 and H7) that prefetches cache lines ahead of the current PC and serves instruction fetches in a single cycle on cache hits. ST's application note AN4566 characterizes the performance impact in detail.

Certain STM32 variants provide a Core Coupled Memory (CCM) region, a block of SRAM connected directly to the D-bus of the processor with zero wait states and no bus contention with DMA or peripheral accesses. On the STM32F4, the 64 kB CCM at 0x10000000 is accessible only by the CPU (not by DMA), making it ideal for the stack, RTOS task stacks, and latency-critical data. Placing performance-critical code in CCM (using the .ccm_text linker section and **attribute**((section(".ccm_text"))) on functions) eliminates both flash wait states and cache misses for that code. This matters most for tight ISRs and PID control loops where cycle-accurate timing is required.

The Cortex-M4 also has an optional instruction cache (I-cache) and data cache (D-cache) on derivatives like the STM32F7 and H7 (which use the M7 core). On these parts enabling the I-cache is almost always a net benefit; enabling the D-cache requires careful attention to cache coherency with DMA, because DMA transfers bypass the cache and can leave the CPU reading stale cached data. The standard pattern is to invalidate (or flush and invalidate) D-cache lines covering a DMA receive buffer before reading, and to clean (write-back) D-cache lines covering a DMA transmit buffer before initiating the transfer.

## Common Misunderstandings

A frequently misunderstood point is the distinction between volatile and atomic. Declaring a variable volatile tells the C compiler to emit a memory access instruction every time the variable is read or written, instead of caching the value in a register across multiple accesses. It does not make any access atomic; a 32-bit load of a volatile uint32_t on a Cortex-M is a single LDR instruction that is atomic on a naturally aligned address, but a 64-bit load of a volatile uint64_t compiles to two LDR instructions, and between them an interrupt can fire and modify the high word, leaving the reader with a torn value. Shared 64-bit counters between an ISR and main code require either a critical section (disable/enable interrupts around the read) or an architecture-specific atomic compare-and-swap (LDREX/STREX on ARMv7-M).

A second misunderstanding involves memory ordering. The Cortex-M3 and M4 (ARMv7-M) have a weakly ordered memory model for device memory but a strongly ordered model for normal memory in the default (MPU-unconfigured) state; however the compiler and the write buffer can still reorder stores to device memory as seen by other bus masters. A peripheral register write followed by a read of a status register from the same peripheral may be reordered if the compiler re-sequences the operations (unless volatile prevents compiler reordering) or if the write buffer has not yet drained when the status read is issued. The solution is a Data Synchronization Barrier (DSB), which stalls the processor until all outstanding memory transactions complete. The Data Memory Barrier (DMB) ensures ordering without stalling (all accesses before the DMB complete before any access after it is observed by other agents). The Instruction Synchronization Barrier (ISB) flushes the pipeline and prefetch buffers, required after writing to the VTOR, enabling the MPU, enabling the FPU, or any operation that changes the processor's instruction fetch context; without an ISB following such changes the processor may fetch and execute instructions based on the old context state.

A third misunderstanding involves the stack pointer alignment requirement. The AAPCS requires the stack pointer to be 8-byte aligned at the point of any public interface call. The Cortex-M hardware enforces 8-byte SP alignment on exception entry by automatically adjusting SP if needed (and recording the adjustment in bit 9 of the stacked xPSR). Hand-written assembly that manipulates SP incorrectly, or C code that uses **attribute**((naked)) functions without managing alignment manually, can violate this requirement and cause hard-to-reproduce data corruption when 64-bit types or NEON/FPU values are involved.

## Key Takeaways

The three-stage in-order pipeline with no dynamic branch prediction means firmware performance is highly predictable and linear: backward branches are cheap, forward branches incur a refill penalty, and placing hot paths in sequential code without branches is the primary throughput optimization available to the firmware engineer without changing the clock.

The distinction between MSP and PSP, and between privileged and unprivileged Thread mode, is the foundation of safe RTOS design on Cortex-M; understanding which stack the processor is using and why prevents the class of bugs where an ISR stack overflow silently corrupts a task's local variables.

The automatic 8-word exception stack frame is the hardware mechanism that allows C functions to serve as ISRs without a wrapper; knowing its layout precisely is essential for fault handlers that walk the stack to report the faulting PC and the register state at the moment of the fault.

Flash wait states, the ART Accelerator, and CCM RAM together determine the real instruction throughput of STM32 firmware at maximum clock; a design that assumes zero-wait-state performance from flash without enabling the accelerator will be 4-5 times slower than expected in instruction-bound code.

The three memory barriers (DSB, DMB, ISB) have distinct and non-interchangeable semantics; using ISB where DSB is needed (or omitting barriers entirely around peripheral register sequences) is a latent correctness bug that may only manifest on higher-speed silicon revisions or after a compiler upgrade that changes instruction scheduling.

## Further Reading

The ARM Architecture Reference Manual for ARMv7-M (document ARM DDI 0403, available from developer.arm.com with a free account) is the definitive normative reference for everything described in this article; chapters B1 (programmer model), B3 (memory model), and B5 (debug) are most directly relevant to firmware engineers. Do not rely on summaries for safety-critical or MISRA-compliant work; the ARM ARM is the authoritative source.

Joseph Yiu's "The Definitive Guide to ARM Cortex-M3 and Cortex-M4 Processors" (3rd edition, Elsevier/Newnes, ISBN 978-0-12-408082-9) is the single best book for practicing firmware engineers; chapters 2-5 cover the programmer model, memory map, exceptions, and fault handling with worked examples from STM32 and LPC hardware. Yiu also wrote "The Definitive Guide to ARM Cortex-M23 and Cortex-M33 Processors" for the ARMv8-M architecture with TrustZone.

ST Microelectronics application note AN4838 "Introduction to STM32 microcontrollers security" covers the MPU configuration and privilege model in the context of STM32L4 and STM32G4, with register-level examples. AN4865 "Getting started with STM32H7 Series hardware development" addresses the cache coherency requirements for the M7 core's D-cache in the context of DMA and Ethernet transfers, and is essential reading for any H7 design using DMA with cached memory regions.

Jacob Beningo's "Reusable Firmware Development" (Apress, ISBN 978-1-4842-3297-2) and accompanying blog series at beningo.com address the API and hardware abstraction patterns that build on the hardware model described in this article; the HAL layer design principles assume the programmer model knowledge developed here. The MISRA C:2025 document and the SEI CERT C Coding Standard (Carnegie Mellon, available at wiki.sei.cmu.edu/confluence/display/c) provide the rule sets most directly applicable to Cortex-M firmware in safety-relevant applications.
