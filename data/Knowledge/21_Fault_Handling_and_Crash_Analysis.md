# Fault Handling and Crash Analysis: Diagnosing Hard Faults on Cortex-M

A production batch of industrial sensor nodes had been running flawlessly on the bench for six weeks. Three days after deployment in the field, units began resetting spontaneously. No error was printed to UART. The watchdog counter in RAM showed a clean value. The firmware developer stared at the code for two days and found nothing. It was only after wiring a JTAG probe to a field unit mid-crash that the answer surfaced: a stacked PC sitting at address 0x00000012, a HardFault caused by a branch-to-NULL through an uninitialized function pointer, and a CFSR reading of 0x00040000 indicating INVPC. Without knowing how to read that information, the fault was invisible. With it, the bug was fixed in twenty minutes.

This is the nature of hard faults on Cortex-M. They are not vague. The CPU records exactly what went wrong, exactly where execution was when it went wrong, and the state of the processor at the moment of failure. The problem is not that the information is unavailable; the problem is that most firmware engineers have never been shown where it lives or how to decode it. A hard fault that resets the chip and clears RAM looks like a mystery. A hard fault where the handler captures the stack frame, reads the fault status registers, and logs them to flash becomes a precise engineering artifact that can be decoded on a desk hours or days after the failure occurred.

This article is a complete reference for fault handling on ARM Cortex-M processors, specifically targeting M3, M4, and M33 variants common in STM32 and similar families. It covers the fault exception hierarchy, every status register and what each bit means, how to write a fault handler that captures actionable data, the most common fault scenarios and how to recognize them, and how to use a debugger and persistent logging to turn a mystery reset into a diagnosed and fixed bug.

## The Cortex-M Fault Hierarchy

The Cortex-M architecture defines a hierarchy of fault exceptions, each targeting a different class of error. At the top sits HardFault, which is always enabled and cannot be disabled. It is the exception of last resort: every other fault that cannot be serviced by its own dedicated handler escalates to HardFault. Below HardFault are three configurable fault exceptions that are disabled by default and must be individually enabled through the System Control Block: MemManage, BusFault, and UsageFault. On Cortex-M33, which implements the TrustZone Security Extension, a fourth configurable fault named SecureFault is added to handle security violations.

MemManage faults are generated by the Memory Protection Unit (MPU). If firmware attempts to access a memory region that the MPU has marked as privileged-only while executing in unprivileged mode, or attempts to execute from a region marked execute-never (XN), or writes to a read-only region, a MemManage fault fires. BusFault is generated by the bus matrix and AHB/APB interconnect when a bus transaction fails. This includes fetching an instruction from an invalid address, reading or writing to a location that produces a bus error response, and errors during exception stack push and pop. UsageFault covers instruction-level errors: undefined instructions, invalid state transitions such as attempting to switch to ARM state which Cortex-M does not support, attempted use of coprocessors that are not enabled, unaligned memory access on targets configured to fault on misalignment, and division by zero when that trap is enabled.

Escalation rules are critical to understand. If MemManage, BusFault, or UsageFault are not individually enabled (the default state after reset), any error that would have generated one of those faults instead escalates to HardFault. If a fault occurs inside another fault handler where the priority would not allow preemption, the fault also escalates to HardFault with a priority violation flag set. On Cortex-M33, if a fault occurs in the Secure state and SecureFault is not enabled, it escalates to HardFault as well. The practical consequence is that on a stock configuration with configurable faults disabled, nearly every error generates a HardFault, and to get the more granular per-type handlers you must enable them by setting the USGFAULTENA, BUSFAULTENA, and MEMFAULTENA bits in SCB->SHCSR at address 0xE000ED24.

## Fault Status Registers

ARM defines a set of registers in the System Control Block (SCB) that capture the cause and address of a fault at the moment it occurs. These registers are memory-mapped into the Private Peripheral Bus (PPB) region starting at 0xE000E000 and are accessible to privileged software at any time. Understanding their locations and contents is the foundation of all fault diagnosis.

The Configurable Fault Status Register (CFSR) lives at address 0xE000ED28 and is 32 bits wide. It is subdivided into three fields: the MemManage Fault Status Register (MMFSR) occupies bits [7:0], the BusFault Status Register (BFSR) occupies bits [15:8], and the UsageFault Status Register (UFSR) occupies bits [31:16]. Each sub-field records the specific type of fault that fired within its category. Bits in CFSR are write-one-to-clear (W1C), meaning a fault handler must read them before clearing them.

The HardFault Status Register (HFSR) lives at address 0xE000ED2C and records why a HardFault occurred. The two diagnostic bits are VECTTBL (bit 1), which indicates a HardFault caused by a bus error on a vector table read during exception processing, and FORCED (bit 30), which indicates that a configurable fault was escalated to HardFault because the configurable exception was disabled or because of a priority problem. DEBUGEVT (bit 31) indicates a debug event. HFSR bits are also W1C.

The Debug Fault Status Register (DFSR) at address 0xE000ED30 records the cause of a debug event (HALTED, BKPT, DWTTRAP, VCATCH, EXTERNAL). This is primarily of interest when stepping through fault handlers under a debugger, but it occasionally captures useful information about breakpoint-triggered faults.

The BusFault Address Register (BFAR) at address 0xE000ED38 holds the address that caused a precise BusFault. It is only valid when BFSR.BFARVALID (bit 15 of CFSR) is set. Similarly, the MemManage Fault Address Register (MMFAR) at address 0xE000ED34 holds the address that caused a precise MemManage fault, and is only valid when MMFSR.MMARVALID (bit 7 of CFSR) is set. Both BFAR and MMFAR are latched at fault time and remain stable until the corresponding valid bit is cleared, but there is a hazard: if a second fault occurs before the first is cleared, the address register may be overwritten and the valid bit cleared. The fault handler must read BFAR and MMFAR before doing anything else that could cause a subsequent fault.

On Cortex-M33, the SecureFault Status Register (SFSR) at address 0xE000EDE4 and the SecureFault Address Register (SFAR) at address 0xE000EDE8 serve the equivalent role for TrustZone security violations. These are only accessible from Secure state.

## Decoding the Cfsr

The CFSR bit field is dense but unambiguous. Breaking it apart by sub-register makes it tractable. Within the BusFault Status Register (BFSR, bits [15:8] of CFSR), the IBUSERR bit (bit 8, CFSR value 0x00000100) indicates an instruction prefetch error where the fetch itself caused a bus error, which typically means the CPU tried to fetch from an address that has no memory or peripheral behind it. PRECISERR (bit 9, value 0x00000200) indicates a precise data bus error: the exact address of the faulting instruction is known and is in the stacked PC, and if BFARVALID is also set, BFAR holds the data address that failed. IMPRECISERR (bit 10, value 0x00000400) indicates an imprecise data bus error where a write buffer in the bus fabric returned an error after the offending instruction had already retired, so the stacked PC points somewhere after the faulting instruction rather than at it; this is the hardest bus fault to diagnose. UNSTKERR (bit 11, value 0x00000800) means a bus fault occurred while unstacking from an exception return. STKERR (bit 12, value 0x00001000) means a bus fault occurred while pushing the exception stack frame. LSPERR (bit 13, value 0x00002000, Cortex-M4/M7 with FPU) means a bus fault occurred during lazy FPU state preservation. BFARVALID (bit 15, value 0x00008000) is the qualifier bit: it is set when BFAR holds a valid fault address.

Within the MemManage Fault Status Register (MMFSR, bits [7:0] of CFSR), IACCVIOL (bit 0, value 0x00000001) indicates an instruction access violation where the CPU tried to fetch an instruction from a region the MPU has marked as non-executable or privileged-only. DACCVIOL (bit 1, value 0x00000002) indicates a data access violation where a read or write was attempted to a region the MPU has disallowed. MUNSTKERR (bit 3, value 0x00000008) means an MPU violation occurred during exception unstack. MSTKERR (bit 4, value 0x00000010) means an MPU violation occurred during exception stack push. MLSPERR (bit 5, value 0x00000020, FPU cores only) means an MPU violation occurred during lazy FPU state save. MMARVALID (bit 7, value 0x00000080) qualifies the MMFAR register.

Within the UsageFault Status Register (UFSR, bits [31:16] of CFSR), UNDEFINSTR (bit 16, value 0x00010000) means the CPU fetched and attempted to decode an instruction with an undefined encoding, which commonly happens when a function pointer is NULL and the CPU fetches from address zero where the vector table encoding is almost certainly not a valid instruction in context. INVSTATE (bit 17, value 0x00020000) means the CPU attempted to execute with an invalid EPSR state, most commonly an attempt to branch to an address with bit 0 clear indicating ARM state which Cortex-M does not support. INVPC (bit 18, value 0x00040000) means the loaded PC value was invalid on an exception return, indicating the EXC_RETURN value on the stack was corrupted. NOCP (bit 19, value 0x00080000) means a coprocessor access was attempted when the coprocessor is absent or disabled, such as using floating-point instructions when the FPU enable bits in the Coprocessor Access Control Register (CPACR at 0xE000ED88) have not been set. UNALIGNED (bit 24, value 0x01000000) indicates an unaligned memory access when the UNALIGN_TRP bit in the Configuration and Control Register (CCR at 0xE000ED14) is set. DIVBYZERO (bit 25, value 0x02000000) indicates an integer division by zero when the DIV_0_TRP bit in CCR is set.

## The Exception Stack Frame

When any exception fires on Cortex-M, the hardware automatically pushes a fixed set of registers onto the active stack before jumping to the exception handler. This automatic state save is what allows the fault handler to inspect the pre-fault machine state without any additional assembly code. The standard (integer-only) stack frame pushed at exception entry consists of eight 32-bit words in this order from the lowest to highest address on the stack: r0, r1, r2, r3, r12, LR (the link register as it was before exception entry), PC (the program counter pointing to the instruction that was executing or about to execute when the fault fired), and xPSR (the program status register, which includes the Thumb bit, condition flags, exception number, and ICI/IT state). The total standard frame is 32 bytes. The stack pointer used is MSP if the fault occurred in Thread mode using MSP or in Handler mode, or PSP if the fault occurred in Thread mode using PSP.

The stacked PC is the most immediately useful field. It is the address of the instruction the processor was about to execute when the fault was taken. For precise faults, this is the exact address of the faulting instruction. For imprecise bus faults (IMPRECISERR), the stacked PC may be several instructions ahead of the actual faulting instruction because the write buffer had already retired the store. To convert the stacked PC to a source line, load the ELF file in GDB and run "list *0xXXXXXXXX" substituting the actual PC value, or use addr2line from the ARM toolchain.

The memory layout of the standard exception frame on the stack is shown below, with lower addresses at the top:

Stack address + 0x00 : stacked r0 Stack address + 0x04 : stacked r1 Stack address + 0x08 : stacked r2 Stack address + 0x0C : stacked r3 Stack address + 0x10 : stacked r12 Stack address + 0x14 : stacked LR (pre-exception link register) Stack address + 0x18 : stacked PC (address of faulting or next instruction) Stack address + 0x1C : stacked xPSR

On Cortex-M4/M7/M33 cores with a floating-point unit, if the Lazy Stack Preservation feature is active (the default), exception entry may push an extended frame that includes FPU registers S0-S15 and FPSCR, adding another 72 bytes to the frame for a total of 104 bytes. The EXC_RETURN value stored in LR at exception entry encodes which frame type was pushed: bit 4 being 0 indicates the extended FP frame was pushed, bit 4 being 1 indicates the standard integer-only frame. The fault handler must account for this when computing the address of the stacked PC by checking EXC_RETURN before indexing into the frame.

## Writing a Hard Fault Handler

A useful fault handler has three responsibilities: capture the stacked frame before anything can corrupt it, read the fault status registers before a secondary fault can clear them, and record everything to a location that survives the reset that will follow. The handler begins in assembly because C cannot reliably access the stack pointer before the C function prologue potentially modifies it.

The assembly trampoline is minimal. On entry to HardFault_Handler, the handler determines which stack was in use by examining bit 2 of the EXC_RETURN value held in LR. If bit 2 is 0, the exception was taken from Handler mode or Thread mode using MSP, so the stacked frame is at MSP. If bit 2 is 1, the exception was taken from Thread mode using PSP, and the frame is at PSP. A minimal GCC-compatible assembly trampoline written for Thumb-2 uses TST LR, #4 to test bit 2, then ITE EQ / MRSEQ R0, MSP / MRSNE R0, PSP to load the correct stack pointer into R0, then branches (not calls) to a C function named fault_handler_c. The branch rather than BL preserves LR (which holds EXC_RETURN) and prevents creating an additional stack frame before the C code runs.

The C diagnostic function receives a pointer to the stacked frame as a uint32_t* argument and reads the stacked values as array elements: index 0 is r0, index 1 is r1, index 2 is r2, index 3 is r3, index 4 is r12, index 5 is the pre-exception LR, index 6 is the stacked PC, and index 7 is the stacked xPSR. The first action in the C function, before any other operation, is to read and store SCB->CFSR (0xE000ED28), SCB->HFSR (0xE000ED2C), SCB->DFSR (0xE000ED30), SCB->MMFAR (0xE000ED34), and SCB->BFAR (0xE000ED38) into local variables declared volatile to prevent the compiler from reordering the reads. These values, along with the stacked PC, stacked LR, and the EXC_RETURN value, are written to the persistent fault record. The CFSR, HFSR, and DFSR registers are then cleared by writing their own values back to them (W1C semantics). The handler ends with either an infinite loop for debug builds (allowing a debugger to attach and inspect the preserved values) or a call to NVIC_SystemReset() for release builds after the log write is confirmed complete.

## Common Fault Scenarios

Null pointer dereference is the most frequent source of HardFaults on Cortex-M. When a pointer holds the value NULL (0x00000000) and code attempts to read or write through it, the CPU issues a bus transaction to address 0x00000000. On STM32, address 0x00000000 aliases to the vector table in flash, and on Cortex-M3/M4 the default bus fabric completes the read without generating a bus error, so you may get DACCVIOL from the MPU (if configured) or garbage data silently. If the NULL pointer is a function pointer and code calls through it, the CPU branches to address 0x00000001 (address zero plus the mandatory Thumb bit), where it finds either an unexpected opcode or an encoding that forms an undefined instruction, producing UNDEFINSTR or INVSTATE in the UFSR. The stacked PC will be 0x00000000 or 0x00000001. Configuring an MPU region covering the first 256 bytes of the address space as no-access converts this silent corruption into a clean MemManage fault with MMFAR showing 0x00000000.

Unaligned access faults fire when UNALIGN_TRP is set in CCR and firmware performs a word or halfword access to a non-aligned address. By default, Cortex-M3/M4 supports unaligned access transparently through hardware splitting, so this fault only fires when you explicitly enable the trap. When it does, UFSR.UNALIGNED is set and the stacked PC points at the misaligned load or store instruction. The most common source is a packed struct whose fields are accessed by dereferencing a byte-aligned pointer cast to a wider type, or a protocol parser that reads multi-byte fields from a byte buffer using a uint32_t pointer.

Stack overflow that destroys the stacked frame is a particularly difficult scenario. When the stack pointer walks past the bottom of the stack allocation and the CPU takes an exception, the hardware tries to push the stack frame to an address below the valid stack region. If that region is unmapped, STKERR fires in BFSR and BFAR may not be valid because the fault was not precise. If the region happens to be mapped to other data (for example the stack overflowed into statically allocated globals), the frame is written on top of variables, corrupting them, and the stacked PC and registers the handler reads may themselves be garbage. Using the MPU to place a no-access guard region immediately below the stack allocation converts this scenario into a clean MemManage fault the instant the stack pointer crosses the boundary, before the overflow can corrupt other data.

Branching to an address with bit 0 clear produces INVSTATE because the Cortex-M architecture interprets bit 0 of a branch target as the Thumb selection bit. A target with bit 0 clear requests a switch to ARM state, which Cortex-M does not support, and INVSTATE fires immediately. This occurs when a function pointer is set to the raw linker address of a function without the Thumb bit, which happens when the address is read from a table, cast from an integer, or computed arithmetically in a way that strips bit 0. The stacked PC is the address of the first instruction the CPU attempted to execute with the incorrect state bit.

Executing data as code produces UNDEFINSTR when the data bytes at the target address do not decode to a valid Thumb instruction. This occurs when an ISR vector table entry points to a data address rather than a function address, when a stack corruption overwrites a return address to point into a data section, or when a DMA misconfiguration overwrites part of the code region. The stacked PC identifies the exact address where the invalid fetch occurred.

## Using a Debugger to Diagnose Faults

When a JTAG or SWD probe is attached and the firmware halts in the fault handler (or a breakpoint is set on HardFault_Handler), the debugger gives direct access to all the information needed for diagnosis. In STM32CubeIDE, the Fault Analyzer view is the fastest entry point: it reads SCB->CFSR, SCB->HFSR, and the stacked frame automatically and presents a decoded summary showing which bits are set and what they mean in plain language. This view is accessible from the Window menu or the Debug perspective during an active debug session on STM32-family targets.

In GDB, the equivalent is performed manually. At the fault handler breakpoint, use "info registers" to see the current register state, then determine whether the stacked frame was on MSP or PSP by inspecting the LR value (EXC_RETURN). If EXC_RETURN bit 2 is 0, the frame is at the current MSP; read the stacked PC with "x /1xw $msp+24". If EXC_RETURN bit 2 is 1, the frame is at PSP; use "p/x $psp" to get the PSP value and read the stacked PC at offset 24 from there. Read the fault registers directly: "x /1xw 0xE000ED28" for CFSR, "x /1xw 0xE000ED2C" for HFSR, "x /1xw 0xE000ED38" for BFAR, and "x /1xw 0xE000ED34" for MMFAR. Once you have the stacked PC, use "info line *0xXXXXXXXX" in GDB or "addr2line -e firmware.elf -a 0xXXXXXXXX" from the toolchain to map it to a source file and line number.

The watch window in Keil MDK and IAR Embedded Workbench exposes the entire SCB peripheral as a structured peripheral register view during debug. Expanding the SCB peripheral shows each CFSR sub-field decoded with bit names, making it possible to identify the fault type without manual bitfield arithmetic. IAR additionally provides an Interrupt Log view that records which exception handlers were entered during the debug session, which is useful for understanding whether a fault escalated from a lower-priority configurable fault handler.

One debugger interaction to be aware of: if the VCATCH bit is set in DFSR (0xE000ED30, bit 3), the fault caused a vector catch debug halt, meaning the debugger stopped the CPU on fault entry before the fault handler ran. Most modern IDEs configure vector catch automatically for HardFault, MemManage, BusFault, and UsageFault, which is useful for live debugging but means the fault handler logging code will not have run. In this state, CFSR holds the fault type but you must manually extract the stacked frame.

## Fault Injection for Testing

Writing a fault handler is not sufficient; the handler itself must be tested. Fault injection means deliberately triggering each class of fault in a controlled test build to verify that the handler fires, captures correct data, and produces a correctly formatted log entry. Testing fault injection paths is part of the definition of done for any embedded system that claims robust fault handling.

For UNDEFINSTR, embed an undefined Thumb instruction encoding using inline assembly: __asm volatile(".short 0xDE00"); produces a permanently undefined 16-bit encoding defined by the ARMv7-M architecture. For DIVBYZERO, set DIV_0_TRP in CCR (bit 4 at 0xE000ED14) and then execute a signed integer divide by zero using a volatile zero divisor to prevent the compiler from optimizing it away. For UNALIGNED, set UNALIGN_TRP in CCR (bit 3) and then perform a word read through a pointer that is offset by one byte from a word-aligned buffer. For a MemManage fault via MPU violation, configure an MPU region as read-only and attempt a write to any address within it, or mark a region as execute-never and branch into it. For STKERR, force a stack overflow in a test task by declaring a local array that exceeds the remaining stack depth by enough to push the SP below the guard region before the exception is taken.

Each injection test should verify not only that the fault handler runs but also that the correct CFSR bits are set, that the stacked PC points to or near the injected fault site (with documented allowance for imprecise faults), and that the persistent log record produced contains accurate data. These tests should be part of the automated test suite executed in a hardware-in-the-loop (HIL) configuration to provide regression coverage.

## Persistent Fault Logging

A fault that occurs in the field and resets the chip leaves no trace unless the firmware is designed to preserve fault information across the reset. The tools available depend on the hardware, but the most common approaches are retained RAM (a region of SRAM backed by the always-on power domain that survives a soft reset), internal flash, and external EEPROM or serial NOR flash.

The retained RAM approach is the fastest and requires no flash write cycles. On STM32 devices, the backup SRAM (BKPSRAM) is in the backup power domain and is preserved through system resets if VBAT is present. On devices without dedicated backup SRAM, a region of regular SRAM can be declared in the linker script as a no-init section using the **attribute**((section(".noinit"))) attribute in GCC to prevent the startup code from zeroing it. The fault handler writes a fixed-size fault record struct to this region immediately on handler entry. The struct should include a magic number to distinguish valid records from uninitialized data, a monotonically increasing fault sequence counter, a reset counter or timestamp, the stacked PC, the stacked LR, the full CFSR value, the HFSR value, the BFAR value, the MMFAR value, and the EXC_RETURN value. On the next boot, early initialization code checks the magic number, forwards the record to the system log (UART, flash log, or over-the-air telemetry), and then clears the record.

For flash-backed logging, the fault handler must use only polled, direct-register-access flash programming routines rather than the HAL flash driver, which may itself have been compromised by the conditions that caused the fault. The flash log region should be a dedicated sector or set of sectors, and each fault record should include a CRC to detect partially written records caused by a power loss during the write. A round-robin wear-leveling scheme across multiple flash slots extends the lifetime of the log region for systems that may fault frequently during development.

The reset cause register (RCC->CSR on STM32L4/F4 families, or the equivalent register on other devices) records the type of reset that occurred: power-on, external NRST pin, software reset, independent watchdog reset, window watchdog reset, or low-power manager reset. Reading and clearing this register at boot and including its value in the fault context provides additional information about whether the reset that followed the fault was triggered by the fault handler calling NVIC_SystemReset (producing a software reset flag) or by the watchdog firing because the fault handler entered an infinite loop or itself crashed.

## The Mpu As a Fault Source

The Memory Protection Unit is both a source of MemManage faults and a powerful tool for converting latent memory corruption bugs into early, clean, diagnosable faults. When the MPU is configured with a region that covers the bottom guard page of a task stack as no-access, any stack overflow that crosses into that region immediately generates a MemManage fault rather than silently corrupting adjacent memory. The MMFSR.MSTKERR bit fires during the exception stacking push when the stack pointer has already crossed the boundary, or MMFSR.DACCVIOL fires if the overflow happened before the exception, with MMFAR pointing to the violated address.

An effective MPU strategy for fault detection uses at minimum one no-access guard region below each task stack, a no-access region over the low addresses of the memory map to catch NULL pointer dereferences, and an execute-never (XN) attribute on all RAM regions where executable code is not intended to reside. The Cortex-M3/M4 MPU supports eight independently configurable regions with base-address and size constraints (size must be a power of two, minimum 32 bytes, base aligned to size). The Cortex-M33 PMSAv8 MPU supports up to 16 regions per security state, with arbitrary base addresses and sizes aligned to 32 bytes, offering significantly more flexible placement.

When diagnosing a MemManage fault triggered by the MPU, the MMFAR value gives you the exact address the access was attempted on, and the stacked PC gives you the exact instruction that made the access. Cross-referencing MMFAR with the linker map file immediately identifies which object occupies that address: its name, section, and size. This is substantially more actionable than discovering the same corruption three function calls later as random data misbehavior, at which point the causal connection to the original access is no longer visible.

## Key Takeaways

Enable the configurable fault exceptions at startup by setting USGFAULTENA, BUSFAULTENA, and MEMFAULTENA in SCB->SHCSR (0xE000ED24). Without these, all faults escalate to HardFault and you lose the per-type granularity the architecture provides, making every fault harder to classify at a glance.

Read CFSR, HFSR, BFAR, and MMFAR as the absolute first operation in your fault handler, using volatile reads, before any C call that could trigger a secondary fault and overwrite the valid-bit qualified address registers.

Write the stacked frame (especially the stacked PC) and all fault status register values to persistent storage before resetting, and verify on the next boot that the record is intact and forwarded to your logging system. A fault that does not produce a retrievable record is a fault you cannot diagnose.

Use the MPU to place no-access guard regions at the bottom of every task stack and over the low address space. This converts stack overflow and NULL pointer writes from silent corruption events into immediate, precisely diagnosed MemManage faults with valid MMFAR values.

Test your fault handler by injecting real faults in controlled test builds, verifying that the correct CFSR bits appear, the stacked PC points to the injection site, and the persistent log record is correctly formatted and readable. A fault handler that has never been exercised under controlled conditions is an untested code path.

## Further Reading

Embedded Artistry maintains a comprehensive series titled "Hard Fault Debugging" at embeddedartistry.com that covers stack unwinding, fault handler assembly patterns, and worked examples of common fault types across different Cortex-M families. The series is freely available and goes deeper into GDB scripting and automated frame extraction than most vendor documentation.

Joseph Yiu's "The Definitive Guide to ARM Cortex-M3 and Cortex-M4 Processors" (third edition, Elsevier) covers the exception model, fault status registers, and MPU in complete detail and is the primary architectural reference for M3/M4 developers. The equivalent reference for M23 and M33 is "The Definitive Guide to ARM Cortex-M23 and Cortex-M33 Processors" by the same author.

STMicroelectronics Application Note AN4830, "Handling of hard fault exceptions on STM32 with IAR, Keil, and GCC compilers," provides a vendor-specific walk-through of fault handler assembly stubs, CFSR decoding, and example code for STM32 families. It is available from the ST website at no cost and is directly applicable to any STM32 project.

The Interrupt blog at interrupt.memfault.com, particularly the article "How to debug a HardFault on an ARM Cortex-M MCU" by Chris Coleman, is the most thorough freely available treatment of the subject. It covers fault handler implementation end to end, provides a complete GDB script for automated frame extraction and CFSR decoding, includes practical examples of real-world fault scenarios with register dumps, and describes integration with the Memfault firmware observability platform for fleet-wide crash reporting.
